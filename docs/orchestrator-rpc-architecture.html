Recommended design for tmux-backed child agents that must be responsive to child questions, blockers, and completions

RPC Orchestrator Architecture

This design keeps your current separate child processes + tmux windows + global registry, but replaces one-way child streaming with a real control plane using pi --mode rpc. The key change is: the coordinator must supervise asynchronously instead of waiting inside one long blocking subagent tool call.

1. System topology

The child runtime is split into two layers: a durable state plane and a live transport plane.

durable state / persistence upward child events downward parent control attention surfaced to coordinator/user
Rule #1: subagent_spawn should launch a tracked child quickly and return. Supervision happens afterward through the registry + RPC hosts, not inside one long-running tool call.

2. Core design rules

1

Separate the planes

RPC is the live transport. SQLite + run dirs are the durable state. tmux is for human visibility, not the source of truth.

2

Never block the coordinator

Do not hold the parent inside a long child-running tool invocation. Spawn, register, and return; then supervise via attention items and injected messages.

3

Normalize every child signal

Questions, blockers, milestones, and completions are converted into agent_messages + agent_events + attention state.

4

Wake the parent intentionally

Idle coordinator: trigger a turn now. Busy coordinator: queue steer or followUp based on urgency.

5

Ack every downward message

Parent reply becomes a registry row first, then is delivered to the child via RPC, then acked after child intake succeeds.

6

Recover on restart

On startup, reconcile tmux, registry, and run dirs; reconnect live RPC hosts where possible; keep unread mailbox state even if no host is attached yet.

3. Main runtime modules

rpc-child-host.ts child-event-normalizer.ts attention-router.ts coordinator-wakeup.ts downward-delivery.ts reconcile.ts
  • RPC Child Host: owns stdio connection to one pi --mode rpc child.
  • Child Event Normalizer: maps raw RPC events into registry updates and previews.
  • Attention Router: decides whether the coordinator, the user, or nobody needs to be interrupted.
  • Coordinator Wake-up Bridge: injects parent messages using sendUserMessage/sendMessage.
  • Downward Delivery: turns a parent decision into child prompt/steer/follow_up RPC commands.
  • Reconcile: restores state after restart and marks lost/stopped/done accurately.

4. Registry contracts

  • agents: current child state, tmux ids, run dir, last preview, final summary.
  • agent_messages: mailbox rows with delivery mode and ack state.
  • agent_events: immutable audit log of lifecycle and transport events.
  • tasks: task-first board remains the thing the coordinator reasons about.
Child signal  → normalize → persist → attention? → wake coordinator?
Parent decision → persist mailbox row → deliver via RPC → ack

5. Delivery policy mapping

  • immediate: send now if the target is idle; otherwise upgrade to steer if urgent.
  • steer: deliver after current turn finishes tool calls.
  • follow_up: deliver after the current run completes.
  • idle_only: store durably, deliver only once target is fully idle.
Urgent child question/blocker   → coordinator: steer
Non-urgent milestone            → coordinator: followUp or no wakeup
Question for user               → user attention item + visible UI
Coordinator answer to child     → child: steer / follow_up per action policy

6. Upward path: child → orchestrator

A

Child publishes proactively

The child should explicitly publish structured signals like question, blocked, or complete. Do not make the parent poll for status.

B

Normalize before waking

Persist first. That way, even if the parent session is restarted, the message survives and can be replayed from the registry.

C

Wake only when necessary

Milestones usually update UI only. Blockers and questions usually wake the coordinator. User-facing questions go to user attention first.

7. Downward path: orchestrator → child

  • Persist before transport. A reply is a mailbox record even if the child is temporarily disconnected.
  • Use action policy. Example: fyi, resume_if_blocked, replan, interrupt_and_replan, stop.
  • Map action policy to RPC delivery. Urgent interrupt → steer. Non-urgent follow-on → follow_up.
  • Ack only after real intake. Don’t mark as delivered just because the parent decided to send it.

8. Coordinator wake-up policy

if child.kind in ["question", "blocked"]:
  if coordinator.isIdle():
    sendUserMessage(renderChildIssue(...))
  else:
    sendUserMessage(renderChildIssue(...), { deliverAs: "steer" })

if child.kind == "question_for_user":
  createUserAttentionItem(...)
  showDashboardBadge(...)
  maybe notify()

if child.kind == "milestone":
  updatePreviewOnly()
  maybe followUp if policy says summarize

if child.kind == "complete":
  addCompletionAttention(...)
  if coordinator.isIdle():
    sendUserMessage(renderCompletionHandoff(...))

9. Child intake policy

switch message.delivery_mode:
  case "immediate":
    if child.isIdle() → RPC prompt
    else → RPC steer
  case "steer":
    RPC steer
  case "follow_up":
    RPC follow_up
  case "idle_only":
    queue until idle, then RPC prompt

10. Human UI surfaces

  • Task board: task is primary object; child links are secondary.
  • Attention queue: blockers, child questions, user questions, completions.
  • Fleet footer/widget: counts of running, blocked, unread, in-review.
  • tmux focus: jump directly into a child window if needed.
  • Manual rescue: stop, capture, reply, reprioritize, reconcile.

11. State machine

launching
  ↓
running
  ├─ child asks question   → waiting / blocked + attention
  ├─ child publishes block → blocked + attention
  ├─ child publishes mile  → running (preview update only)
  ├─ parent sends cancel   → stopped
  ├─ child finishes        → done + completion attention
  └─ transport disappears  → lost (until reconcile proves otherwise)

waiting/blocked
  ├─ parent answer delivered + child resumes → running
  ├─ user answer delivered + child resumes   → running
  └─ parent cancels                          → stopped

State comes from normalized events, not raw tmux transcript scraping.

12. Restart and recovery

1

Load registry

Rehydrate agents, unread messages, and unresolved attention items.

2

Reconcile tmux

Check whether expected tmux windows still exist and whether the child process is still alive.

3

Reconnect RPC where possible

If the child launcher supports host reattachment, restore the live control plane; otherwise keep durable mailbox state and use focus/manual tools.

4

Replay unread coordinator work

Re-surface unresolved child questions/completions to the coordinator/user through the attention queue.

14. The single most important architecture decision

Do not make the coordinator wait synchronously on child execution.
The child must become a tracked background actor. The coordinator should react to child mailbox events through attention routing and injected messages, not sit trapped inside one long-running tool call waiting for the child to end.

This design matches Pi’s documented primitives: child transport via RPC mode, event observation via streamed events, coordinator wake-up via sendMessage/sendUserMessage, queue semantics via steer/followUp, and durable replay via session/registry state.
